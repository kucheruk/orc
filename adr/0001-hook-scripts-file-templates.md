# ADR-0001: Хранение hook-скриптов как файловых шаблонов

## Статус
Принято (2026-03-01)

## Контекст
- В `orc_core/hooks.py` hook runtime (`orc_hook_lib`, `orc_before_submit`, `orc_stop`) хранился как длинные string literals.
- Такой подход ухудшает DevEx: нет полноценного линтинга/подсказок/типизации для кода внутри строк.
- Ошибки синтаксиса в строках проявляются только во время runtime hook execution.
- Дублировались части логики и усложнялась поддержка.

## Решение
- Канонический источник hook runtime перенесён в реальные Python-файлы:
  - `orc_core/hook_scripts/orc_hook_lib.py`
  - `orc_core/hook_scripts/orc_before_submit.py`
  - `orc_core/hook_scripts/orc_stop.py`
- `ensure_repo_hooks()` в `orc_core/hooks.py` теперь:
  1. читает шаблоны из `orc_core/hook_scripts/`,
  2. делает placeholder-injection (`__ORC_ROOT__` -> путь до ORC root),
  3. копирует рендер в `<workspace>/.cursor/hooks/*.py`,
  4. выставляет executable mode (`chmod 0o755`).
- По умолчанию запись идемпотентна: файл перезаписывается только если содержимое изменилось или файла ещё нет.
- Для `--reinit-hooks` добавлен `force_rewrite=True`, который принудительно пересоздаёт hook-файлы.

## Последствия

### Положительные
- Полноценная поддержка инструментов разработки (lint, IDE, type checks) для hook-кода.
- Снижается риск runtime-only синтаксических ошибок.
- Архитектура проще: меньше string-инжиниринга, проще ревью и сопровождение.
- Идемпотентная генерация уменьшает лишние перезаписи в обычном цикле.

### Негативные/компромиссы
- Появляется дополнительная директория с шаблонами, которую нужно держать в актуальном состоянии.
- Появляется явный контракт placeholder-ов (`__ORC_ROOT__`), требующий дисциплины при изменениях.

## Альтернативы
- Оставить string literals в `hooks.py` — отклонено из-за плохой поддерживаемости.
- Использовать внешние `.tmpl` вне модуля `orc_core` — отклонено, т.к. канонический runtime должен жить рядом с кодом генератора.

## Инварианты после решения
- Hook scripts исполняются из `<workspace>/.cursor/hooks/*.py`.
- `ensure_repo_hooks_config()` продолжает писать команды вида `python3 <hook_path>`.
- `--reinit-hooks` остаётся явным механизмом принудительной переинициализации.
