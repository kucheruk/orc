# ADR-0002: Команды repo-hooks как исполняемые скрипты

## Статус
Принято (2026-03-01)

## Контекст
- ORC генерирует hook-скрипты в `<repo>/.cursor/hooks/*.py` с shebang `#!/usr/bin/env python3`.
- В `ensure_repo_hooks()` для этих скриптов уже выставляется executable mode (`chmod 0o755`).
- При этом `ensure_repo_hooks_config()` ранее писал в `.cursor/hooks.json` команды вида `python3 <hook_path>`.
- Такой контракт расходился с инвариантом "скрипт сам является исполняемым бинарником" и усложнял переносимость поведения.

## Решение
- В `.cursor/hooks.json` использовать прямые команды запуска скриптов:
  - `<repo>/.cursor/hooks/orc_before_submit.py`
  - `<repo>/.cursor/hooks/orc_stop.py`
- Добавить безопасную миграцию legacy-формата:
  - если в `beforeSubmitPrompt`/`stop` есть точная команда `python3 <тот же абсолютный путь>`, она заменяется на новый формат;
  - миграция идемпотентна и не создает дубли.
- Пользовательские кастомные команды, не совпадающие с точным legacy-шаблоном, не переписываются.

## Последствия

### Положительные
- Единый и непротиворечивый контракт: shebang + executable bit + прямой вызов.
- Меньше риска двойных/дублированных записей при обновлении конфигурации.
- Конфиг проще читать и диагностировать.

### Компромиссы
- На Windows POSIX execute bit не используется, но формат команды остается единым для всех платформ.
- Корректность запуска по-прежнему зависит от корректного shebang в сгенерированных скриптах.

## Инварианты после решения
- ORC изменяет только repo-local hooks (`<repo>/.cursor/hooks.json`).
- `ensure_repo_hooks_config()` пишет команды прямого вызова hook-скриптов.
- Повторные вызовы генерации не приводят к накоплению дублей в `beforeSubmitPrompt` и `stop`.
