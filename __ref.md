# Отчёт по зонам рефакторинга (`ORC-SMOKE-001`)

## Контекст и метод

- Источники правды: `AGENTS.md`, `techspec.md`, кодовая база (`orc.py`, `orc_core/*`, `.cursor/hooks/*`).
- Формат анализа: зона -> симптомы -> риск -> приоритет -> безопасный план.
- Цель: определить минимально рискованный порядок улучшений без регрессий в hook-driven completion.

## Ключевые выводы

1. В проекте есть архитектурный долг по границам модулей: оркестрация, commit-фаза, retry-логика и fallback-логика находятся в одном крупном контуре.
2. Есть дублирование бизнес-правил между `orc_core/backlog.py` и встраиваемым `orc_hook_lib` внутри `orc_core/hooks.py`.
3. Генерация hook-скриптов "строками" повышает риск рассинхронизации и усложняет тестирование.
4. Тестовый контур отсутствует (нет `tests/`), поэтому инварианты из `techspec.md` не защищены автопроверками.
5. Есть расхождение с `techspec.md` в тексте follow-up сообщения stop-hook (контракт выходного JSON не полностью совпадает).

## Приоритизированные зоны рефакторинга

### P0 — Контракт task/backlog parser в одном месте

- **Где:** `orc_core/backlog.py`, `orc_core/hooks.py` (embedded `orc_hook_lib`).
- **Симптомы:** повторяются `TASK_RE`, `TASK_ID_RE`, логика извлечения ID и пометки `[x]`.
- **Риск:** разъезд regex/правил маркировки приведёт к зависшим задачам и некорректному завершению через hooks.
- **Почему P0:** это базовый инвариант из `techspec.md` (поиск и mark task по ID).
- **План:**
  - вынести парсер/контракт task line в отдельный модуль уровня `orc_core/task_contract.py`;
  - использовать его и в оркестраторе, и в hook runtime;
  - покрыть golden-тестами форматы строк (`- [ ]`, `- [x]`, `**ID:**` и т.д.).

### P0 — Декомпозиция `orc_core/supervisor.py`

- **Где:** `orc_core/supervisor.py`.
- **Симптомы:** файл ~1000+ строк, смешаны CLI, lifecycle state machine, fallback invoke, commit-phase, git operations, cleanup.
- **Риск:** высокая вероятность регрессий при локальных изменениях; трудно локализовать дефект по логам.
- **Почему P0:** это главный runtime-контур, через него проходит весь жизненный цикл задачи.
- **План:**
  - разделить на модули: `cli_config`, `task_lifecycle`, `commit_phase`, `fallbacks`, `finalizers`;
  - сохранить существующие публичные аргументы CLI без изменений;
  - добавить thin orchestrator в `main()` с явными шагами state machine.

### P1 — Убрать встраивание hook-кода как больших строк

- **Где:** `orc_core/hooks.py` (`ensure_repo_hooks`).
- **Симптомы:** содержимое `orc_before_submit.py`, `orc_stop.py`, `orc_hook_lib.py` зашито в Python-строки.
- **Риск:** невозможно нормально линтить/тестировать hook-логику до генерации; редактирование рискованно.
- **Почему P1:** влияет на поддерживаемость и скорость исправлений, но не блокирует запуск прямо сейчас.
- **План:**
  - перенести шаблоны hooks в отдельные файлы (например, `templates/hooks/*.py.tpl`) или реальные модули с копированием;
  - ввести smoke-тест генерации/синтаксиса для output hook files;
  - зафиксировать версионирование шаблонов.

### P1 — Единый модуль уведомлений (Telegram)

- **Где:** `orc_core/notify.py` и embedded `send_telegram_message` в `orc_hook_lib`.
- **Симптомы:** дублируются чтение конфигов, truncate, сетевой вызов, логирование.
- **Риск:** при изменении API/валидации сообщения часть runtime останется на старой логике.
- **План:**
  - вынести общий функционал в shared-runtime helper, доступный и оркестратору, и hooks;
  - определить единый policy для ошибок отправки (не валим процесс, но пишем диагностические поля).

### P1 — Сверка с `techspec.md` по follow-up контракту stop-hook

- **Где:** embedded `orc_stop.py` в `orc_core/hooks.py`.
- **Симптомы:** в stdout выдаётся `commit EVERYTHING+push ...`, тогда как в `techspec.md` описано `commit+push with task ID and task description as commit message`.
- **Риск:** внешние автоматизации, полагающиеся на фиксированный follow-up контракт, могут интерпретировать событие неверно.
- **План:**
  - либо привести код к спецификации,
  - либо обновить `techspec.md`, если принято новое контрактное поведение.

### P2 — Очистка/консолидация вспомогательных модулей

- **Где:** `orc_core/monitor.py`, частично `orc_core/text_parse.py`.
- **Симптомы:** `monitor.py` по сути shim-реэкспорт; в `text_parse.py` есть жёстко заданные маркеры модели/интерфейса.
- **Риск:** невысокий, но создаёт путаницу и hidden coupling к конкретной версии UI Cursor.
- **План:**
  - решить судьбу `monitor.py` (оставить как compatibility layer и документировать или удалить после проверки внешних импортов);
  - сделать parsing status lines менее завязанным на конкретное имя модели.
