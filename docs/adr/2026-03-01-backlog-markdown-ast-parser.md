# ADR: AST-парсинг BACKLOG через markdown-it-py

## Статус
Принято (2026-03-01)

## Контекст
- Парсинг задач в backlog опирался на построчное regex-сопоставление.
- При вариативном Markdown (ссылки в ID, вложенные списки, нестандартные отступы) задачи могли не распознаваться.
- Оркестратор зависит от корректного выделения `task_id` и статуса чекбокса; пропуск задачи нарушает наблюдаемость и управляемость потока.

## Решение
- Перевести извлечение задач на AST-подход с `markdown-it-py`:
  - разбирать весь markdown-документ в токены;
  - обрабатывать `list_item` узлы;
  - извлекать чекбокс из inline-содержимого пункта списка;
  - извлекать `task_id` из нормализованного текста пункта.
- Для `mark_task_done(task_id)` обновлять только маркер чекбокса в исходной строке списка, не меняя описание задачи.
- Сохранить текущую семантику по дубликатам ID: если встречаются несколько пунктов с одним `task_id`, открытые копии помечаются как выполненные.

## Альтернативы
- Усилить текущие regex — отклонено: сложность растёт, но остаются слепые зоны markdown-контекста.
- Гибридный режим (regex + AST fallback) — отклонено: дублирование логики и неоднозначное поведение.

## Последствия

### Положительные
- Парсер устойчив к форматированию Markdown, включая ссылочный ID и вложенные чеклисты.
- Линии, похожие на задачи внутри code fence, не попадают в backlog-задачи.
- Снижен риск «пропажи» задач из поля зрения оркестратора.

### Компромиссы
- Добавлена прямая зависимость от `markdown-it-py`.
- Логика парсинга стала структурно сложнее, чем один regex, но предсказуемее в реальных markdown-сценариях.

## Инварианты после решения
- Матчинг и завершение задач продолжают работать строго по `task_id`.
- `mark_task_done` остаётся идемпотентным и не редактирует текст задачи.
- Hook/runtime поведение не меняется: меняется только способ извлечения задач из backlog.
