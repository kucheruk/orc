# ADR: устойчивый детект waiting_for_input через структурные сигналы и конфиг

## Статус
Принято (2026-03-01)

## Контекст
- В `StreamJsonMonitor._is_followup_prompt_event` детект ожидания пользовательского ввода строился на наборе захардкоженных строк в человекочитаемом тексте `result/error`.
- Такой подход хрупок: при изменении формулировок Cursor CLI или локали оркестратор может не распознать `waiting_for_input` и перейти в `stalled`.
- Требуется сохранить совместимость, но убрать зависимость от конкретных UI-фраз.

## Решение
- Вынести логику детекта в отдельный модуль `orc_core/followup_detection.py`.
- Ввести двухуровневую стратегию:
  1. **Структурные сигналы (приоритетно):**
     - распознавание известных `subtype` (например, `user_input_requested`);
     - распознавание truthy-флагов запроса ввода в структуре события (`requires_user_input`, `waiting_for_input` и т.п.).
  2. **Текстовые маркеры (резервный путь):**
     - проверка по маркерам в сыром event JSON только для `result` с ошибочным статусом.
- Перенести список маркеров и статус-гейтов в проектный конфиг `followup_markers.json` в корне репозитория.
- Если конфиг отсутствует или повреждён, использовать встроенный безопасный набор значений по умолчанию.

## Последствия
### Положительные
- Детект меньше зависит от изменений UI-текста в Cursor CLI.
- Настройка маркеров возможна без правки кода оркестратора.
- `StreamJsonMonitor` стал проще: оркестрация и детект разделены.

### Компромиссы
- Появляется отдельный конфиг-контракт (`followup_markers.json`), который нужно держать актуальным.
- Риск ложных срабатываний по структурным флагам контролируется ограниченным списком ключей и строгой truthy-проверкой.

## Инварианты
- Контракты lifecycle (`waiting_for_input` / `stalled` / `completed`) не меняются.
- Поведение `ui_followup_prompt` остаётся sticky на время жизни монитора.
- Детект выполняется в контуре stream-json и не требует терминального scraping.
