# ADR: CLI entry point и packaging для ORC

## Контекст

Запуск ORC требовал запоминать путь до `orc.py` и выполнять команду вида:

```bash
uv run python /path/to/orc/orc.py --workspace /path/to/repo
```

Это ухудшало DevEx:
- запуск зависел от ручного указания пути;
- требовались алиасы/локальные обертки;
- существующий `pyproject.toml` не использовался как источник CLI-интеграции.

Также проект использует модуль `orc_core`, что требует явной конфигурации packaging для стабильной сборки/install.

## Решение

Принято сделать ORC installable CLI-утилитой:

1. Добавить в `pyproject.toml` entry point:
   - `[project.scripts]`
   - `orc = "orc_core.cli_app:main"`
2. Включить packaging-режим для uv:
   - `[tool.uv] package = true`
3. Явно зафиксировать build backend и состав wheel-пакета:
   - `hatchling` как `build-backend`;
   - `packages = ["orc_core"]` в target wheel.

Целевой UX:
- установка через `uv tool install /path/to/orc` (или `pipx install /path/to/orc`);
- запуск из любой директории через `orc` без знания пути к `orc.py`.

## Альтернативы

1. Оставить запуск только через `uv run python .../orc.py`.
   - Плюсы: ноль изменений в packaging.
   - Минусы: плохой UX запуска, лишняя когнитивная нагрузка.

2. Сделать shell-обертки/алиасы и оставить проект не-installable.
   - Плюсы: быстрый локальный workaround.
   - Минусы: отсутствие единого стандарта запуска, проблемы переносимости между окружениями.

3. Использовать installable CLI через `pyproject` (выбрано).
   - Плюсы: стандартный путь установки и запуска, предсказуемый DX, совместимость с `uv tool`/`pipx`.
   - Минусы: требуется поддерживать packaging-конфигурацию.

## Последствия

- ORC запускается командой `orc`, что устраняет UX-слепую зону старта.
- `pyproject.toml` становится источником истины для CLI-интеграции.
- Документация и runbook переводятся на новый стандарт запуска.
- Fallback-запуск через `orc.py` не поддерживается, чтобы исключить обход стандартного CLI-пути и держать поведение единообразным.
