# ADR: Post-mortem экран для фатальных завершений TUI

## Контекст

Основной экран выполнения ORC в Textual (`Reasoning` + `Event Feed`) закрывался сразу после завершения процесса.

При фатальных исходах (`max_restarts_exceeded`, `commit_phase_failed`, необработанные исключения) это создавало слепую зону наблюдаемости:

1. Последние шаги reasoning исчезали мгновенно.
2. Разработчик был вынужден сразу переходить к большим debug-логам.
3. Усложнялся быстрый разбор инцидента "на месте" в UI.

Нужно сохранить простой UX на успехе и добавить явный post-mortem режим только для аварийных сценариев.

## Решение

Выбран отдельный post-mortem экран в Textual:

- в `OrcApp.on_orchestrator_finished` при фатальном завершении вместо `self.exit(...)` вызывается `push_screen(PostMortemScreen(...))`;
- экран показывает красный баннер с причиной падения, последние `Reasoning`/`Event Feed` и базовую диагностику (task id, exit code, debug log);
- выход из post-mortem происходит только по `Esc`, после чего приложение завершается с исходным кодом ошибки;
- причина фейла (`failure_reason`) прокидывается в событие `OrchestratorFinished` из runtime-слоя.

Критерии фатальности:

- всегда фатально: наличие `error_text` (необработанное исключение);
- фатально: любой `exit_code != 0`, кроме `130` (ручная остановка);
- явно выделены ожидаемые причины: `max_restarts_exceeded`, `commit_phase_failed`.

## Альтернативы

1. Оставить текущее немедленное закрытие.
   - Плюсы: минимальный код.
   - Минусы: потеря контекста прямо в момент падения.

2. Писать только в лог и показывать путь к файлу.
   - Плюсы: без изменений TUI-flow.
   - Минусы: медленный triage, высокий когнитивный шум.

3. Отдельный post-mortem экран (выбран).
   - Плюсы: быстрый локальный разбор, прозрачный UX, минимальные изменения архитектуры.
   - Минусы: дополнительная ветка UI-состояния.

## Последствия

- Улучшена наблюдаемость аварийных сценариев без изменения успешного happy-path.
- Ускорен первичный triage падений прямо в TUI.
- Добавлены тесты на post-mortem flow и выход по `Esc`.
- Поведение ручной остановки (`130`) сохранено без лишних экранов.
