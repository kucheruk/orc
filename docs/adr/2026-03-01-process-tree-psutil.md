# ADR: Кроссплатформенное управление деревом процессов через psutil

## Контекст

В `orc_core/process.py` управление дочерними процессами опиралось на shell-вызов `pgrep -P` и POSIX-сигналы через `os.kill(..., SIGTERM/SIGKILL)`.

Это создавало два системных риска:

1. Непереносимость на Windows и хрупкость в окружениях без `pgrep`/`procps`.
2. Нестабильность при гонках жизненного цикла процессов (процесс мог исчезнуть между шагами обработки).

Нужно сохранить текущий контракт вызовов и повысить надежность без усложнения оркестратора.

## Решение

Выбран переход на `psutil` для работы с деревом процессов:

- получение потомков через `psutil.Process(pid).children(...)`;
- завершение через `terminate()`;
- ожидание через `psutil.wait_procs(...)`;
- эскалация на `kill()` только для оставшихся живыми;
- корректная обработка `NoSuchProcess`, `ZombieProcess`, `AccessDenied` без падения оркестратора.

Контракт `kill_process_tree(root_pid, log_path, label)` сохранен; изменена только внутренняя реализация.

## Альтернативы

1. Оставить `pgrep` + `os.kill`.
   - Плюсы: без новой зависимости.
   - Минусы: OS-зависимость, медленнее и хрупче из-за shell-процесса.

2. Реализовать платформенные ветки вручную (`ps`/WMI/ctypes и т.д.).
   - Плюсы: без внешней библиотеки.
   - Минусы: высокая сложность поддержки и риск несовместимостей.

3. Переход на `psutil` (выбран).
   - Плюсы: кроссплатформенность, зрелый API, меньше race-condition в коде.
   - Минусы: новая зависимость.

## Последствия

- Устранена runtime-зависимость от `pgrep` и POSIX-only механики.
- Повышена устойчивость процесса завершения дерева на разных ОС.
- Добавлена зависимость `psutil` в проект.
- Требуется поддерживать тесты на edge-case сценарии (`NoSuchProcess`/`AccessDenied`) при будущих изменениях process lifecycle.
