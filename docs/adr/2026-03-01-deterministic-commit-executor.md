# ADR: Детерминированный commit executor в ORC

## Статус
Принято

## Контекст

Ранее commit phase делегировала выполнение `git commit` самому агенту через текстовый промпт.  
Это создавало системный риск:
- недетерминированные ошибки экранирования/кавычек в shell-командах;
- нестабильное поведение при спецсимволах в сообщении коммита;
- потенциальная попытка выполнения лишних git-команд, не относящихся к задаче.

Для оркестратора это "слепая зона надежности": критичная мутация репозитория зависела от LLM-вывода, а не от контролируемого кода runtime.

## Решение

1. В commit phase агент **только генерирует commit message**.
2. Формат ответа: только строгий JSON (`title`, `body`) без fallback.
3. ORC сам выполняет git-мутации через subprocess:
   - `git add -A`
   - `git diff --cached --quiet`
   - `git commit -m <title> [-m <body>]`
4. Если сообщение невалидно или git-команда завершается ошибкой — fail-fast.
5. Требование к заголовку: начинается с `TASK-ID:`.

## Альтернативы

### A1. Оставить старую модель (LLM сам выполняет git)
- Плюсы: проще промпт на стороне ORC.
- Минусы: высокая вероятность недетерминированных сбоев в shell-синтаксисе и низкая гарантия корректного исполнения.

### A2. Разрешить JSON + plain text fallback
- Плюсы: потенциально меньше fail-fast при нестабильном выводе.
- Минусы: размывает контракт, усложняет валидацию, увеличивает риск неоднозначного парсинга.

### A3. Только JSON без fallback (выбран)
- Плюсы: строгий и однозначный контракт, предсказуемый парсинг, меньше скрытых веток.
- Минусы: больше fail-fast при плохом ответе агента, требует дисциплины промпта.

Выбрано: строгий JSON-only контракт как приоритет надежности и детерминизма.

## Последствия

- Появляется детерминированный контур выполнения коммита в runtime ORC.
- Уменьшается зависимость от shell-генерации агентом и ошибки экранирования.
- Диагностика становится проще: видно шаг отказа (`add_all` / `cached_quiet` / `commit`).
- Commit phase становится быстрее и стабильнее, так как LLM не тратит шаги на написание bash-команд.
