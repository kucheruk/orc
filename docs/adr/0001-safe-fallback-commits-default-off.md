# ADR-0001: Безопасный fallback commit — только opt-in

## Статус
Принято

## Контекст

В commit phase использовался опасный fallback: при неудачном коммите оркестратор мог выполнить `git add -A` и сделать checkpoint-коммит автоматически.  
Это создает риск попадания в историю репозитория:
- секретов (`.env.local`, ключи, временные дампы),
- промежуточных/битых изменений, не прошедших проверки,
- нерелевантных файлов, которые агент не должен был коммитить.

Такое поведение подрывает доверие к автоматизации и усложняет восстановление корректного состояния ветки.

## Решение

1. Fallback-коммит в commit phase делаем явным opt-in.
2. Добавляем флаг `--allow-fallback-commits`.
3. Значение по умолчанию: `False` (fallback выключен).
4. Если commit phase оставляет tracked-изменения и fallback выключен, ORC завершает задачу ошибкой (fail-fast) и оставляет изменения в working tree для ручного решения человеком.

## Альтернативы

### A1. Оставить безусловный fallback (старое поведение)
- Плюсы: меньше ручных действий в коротком горизонте.
- Минусы: высокий риск автокоммита мусора/секретов, потеря контроля и доверия.

### A2. Частичный fallback с whitelist/blacklist файлов
- Плюсы: потенциально безопаснее безусловного `git add -A`.
- Минусы: сложная поддержка правил, высокий риск ложных допусков/блокировок, рост сложности.

### A3. Полностью убрать fallback
- Плюсы: максимально предсказуемо.
- Минусы: теряется полезный аварийный инструмент для опытного оператора в отдельных сценариях.

Выбрано решение opt-in как баланс безопасности и операционной гибкости.

## Последствия

- По умолчанию поведение стало безопаснее и предсказуемее: нет скрытых автокоммитов.
- Оператор чаще принимает осознанное решение по `git diff`/`git status`.
- При необходимости fallback доступен явно через флаг и воспринимается как осознанный риск.
