# ADR 0001: Атомарная запись файлов состояния

## Контекст

В ORC несколько процессов (оркестратор и hook-скрипты Cursor) читают и обновляют одни и те же файлы состояния, в первую очередь `.cursor/orc-task.json`.
До изменения запись выполнялась через `Path.write_text(...)`, что делало окно для race condition:

- один процесс мог прочитать файл в момент truncate/rewrite;
- при одновременных update возникал риск битого JSON и плавающих падений на decode;
- аналогичная проблема затрагивала и другие state-файлы (`.cursor/hooks.json`, `.orc/orc-metrics.json`, `.orc/orc-stats.json`, `.orc/model-selection.json`, `.orc/orc.lock`) и перезаписываемые prompt-файлы.

С учётом требования идемпотентности и надёжности нужно убрать неатомарные перезаписи как класс.

## Решение

Введён единый модуль `orc_core/atomic_io.py`:

- `write_text_atomic(path, text, encoding="utf-8")`
- `write_json_atomic(path, payload, ensure_ascii=False, indent=2)`

Алгоритм записи:

1. создать временный файл в той же директории;
2. записать данные и выполнить `flush + fsync`;
3. выполнить `os.replace(tmp, target)` как атомарную замену;
4. выполнить best-effort `fsync` директории;
5. удалить временный файл при исключении.

На этот механизм переведены все production-точки перезаписи файлов в `orc_core/*`, включая генерацию hook-файлов и встроенный runtime `orc_hook_lib`.

## Альтернативы

1. **Оставить `write_text` и полагаться на fallback при чтении битого JSON**  
   Отклонено: это маскирует причину, а не устраняет её.

2. **Добавить файловые lock-и поверх текущей записи**  
   Частично решает lost-update, но не устраняет риск partial-write при падении процесса и усложняет код.

3. **Использовать SQLite/другой storage вместо файлов**  
   Избыточно для текущего объёма данных и нарушает KISS для этой задачи.

## Последствия

Позитивные:

- резко снижается вероятность corruption для state-файлов;
- единый helper уменьшает дублирование и упрощает сопровождение;
- поведение записи становится одинаковым в runtime и hook-шаблонах.

Компромиссы:

- небольшая стоимость на `fsync` при каждой записи;
- lock-файл всё ещё имеет отдельный TOCTOU-аспект на этапе `exists/unlink/write` (это отдельный follow-up по синхронизации захвата, не по атомарности перезаписи).
