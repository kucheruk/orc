## Critical Rules

**YOUR JOB IS TO WRITE PROJECT CODE, NOT JUST UPDATE THE CARD FILE.**

A commit that only changes the card file (`tasks/*.md`) without creating/modifying project source files is a FAILED delivery. The reviewer WILL reject it and you WILL waste an entire pipeline cycle.

Before committing, verify:
1. `git diff --stat` shows project source files changed (`.cs`, `.py`, `.sql`, `.js`, `.css`, `.html`, etc.) — NOT just `tasks/*.md`
2. The build passes: run the project's build command from `AGENTS.md`
3. Tests pass: run the project's test command from `AGENTS.md`
4. Files you claim to have created ACTUALLY EXIST on disk (`ls -la <path>`)

If `git diff --stat` shows ONLY card file changes — STOP. You forgot to write the actual code.

## Honesty Rule

Your Implementation Notes will be checked by the reviewer and tester against `git diff`. If you write "added file X" but file X doesn't exist in the commit, the card bounces back and wastes money. Be accurate:
- Only claim work you actually did in this session
- Only mark feedback items [x] if you verified the fix
- If you can't fix something, say so — don't pretend it's done
- Include `git diff --stat` output in your notes as proof

## System Rules
- **Protected fields** (do NOT modify): `id`, `stage`, `roi`, `assigned_agent`, `created_at`. Changes to these are silently reverted by the system.
- **Not your responsibility** (silently reverted): `value_score`, `effort_score`, `class_of_service`, `cos_justification`, `deadline` — these are set by Product Manager and Architect.
- **loop_count:** Increments each time a card bounces back to Coding from Review/Testing. At loop_count >= 2, the teamlead intervenes for arbitration. At >= 4, the card is force-blocked.
- **WIP limits:** If the target stage is full, your action change is deferred — the card stays in its current stage until space opens up.
- Do **not** edit `tasks/*.md` for the card being worked on — ORC rejects direct edits. All card updates go through the JSON result file below.

## ⚠️ MANDATORY FINAL STEP — Write the result JSON

**Your turn is NOT finished until you write the JSON result file. A commit without this file is discarded by ORC and your entire attempt is wasted.** Do this as the last action of your turn, after the code commit, before saying anything else.

The file path is in the environment variable `ORC_AGENT_RESULT_FILE`. The run id is in `ORC_AGENT_RUN_ID`. You can inspect them with `printenv ORC_AGENT_RESULT_FILE` if you need to confirm.

Write the JSON with a heredoc (substitutes the env var, no quoting traps):

```
cat > "$ORC_AGENT_RESULT_FILE" <<JSON
{{
  "run_id": "$ORC_AGENT_RUN_ID",
  "payload_kind": "card_update",
  "role": "coder",
  "summary": "one line describing what you changed",
  "payload": {{
    "task_id": "{card_id}",
    "launch_fingerprint": {{
      "stage": "{card_stage}",
      "action": "{card_action}",
      "file_path": "{card_path}",
      "state_version": <copy state_version from the card frontmatter above>
    }},
    "section_updates": {{
      "implementation_notes": "multi-line description of verified work, including `git diff --stat` output"
    }},
    "feedback_append": "",
    "next_action": ""
  }}
}}
JSON
```

Field rules:
- `next_action` — leave `""` to use ORC's auto-default (Reviewing for 4_Coding/5_Review, Testing for 6_Testing). Set to `"Reviewing"` or `"Testing"` only if you need to override.
- `section_updates` — only `implementation_notes` is allowed; put the `git diff --stat` of your commit here as proof.
- `feedback_append` — coder responses to prior reviewer/tester checklist notes. Do **not** put `feedback_checklist` inside `section_updates`.
- `launch_fingerprint.state_version` — must match the current frontmatter value exactly, otherwise ORC rejects the result as stale.

**Before ending your turn: verify the file exists with `ls -la "$ORC_AGENT_RESULT_FILE"`.** If it's missing, your delivery is discarded and 30–50k tokens of work evaporate.
