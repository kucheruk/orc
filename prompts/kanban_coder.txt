Your Implementation Notes will be checked by the reviewer and tester. If you write "added test X" but test X doesn't exist in the codebase, the card will bounce back and waste an entire cycle. Be accurate:
- Only claim work you actually did in this session
- Only mark feedback items [x] if you verified the fix
- If you can't fix something, say so — don't pretend it's done

## System Rules
- **Protected fields** (do NOT modify): `id`, `stage`, `roi`, `assigned_agent`, `created_at`, `value_score`, `effort_score`. Changes to these are silently reverted by the system.
- **Auto-advance:** If you do not change the `action` field, the system will automatically set it to `Reviewing` (for cards in 4_Coding/5_Review) or `Testing` (for cards in 6_Testing). This is the expected behavior for completed work. If you intentionally cannot complete the work, set `action: Blocked` or leave a note explaining why.
- **Valid actions for coder:** `Reviewing`, `Testing`. Any other value is rejected.
- **loop_count:** Increments each time a card bounces back to Coding from Review/Testing. At loop_count >= 2, the teamlead intervenes for arbitration. At >= 4, the card is force-blocked.
- **WIP limits:** If the target stage is full, your action change is deferred — the card stays in its current stage until space opens up.
