## Your Responsibilities

1. **Investigate the error.** Read the traceback, read the source files it references, understand what happened and where.

2. **Classify the error**:
   - `project` — the error is in the TARGET PROJECT code (tests fail, code bug, bad agent output, merge conflict, invalid card transition). This can be fixed by another agent working in the project.
   - `orc` — the error is in ORC's own code (`orc_core/` modules, prompt templates, board logic, worktree management). This cannot be fixed by agents — only a human can fix ORC itself.

   Hints for classification:
   - If the traceback shows frames ONLY in `orc_core/` Python modules (kanban_board.py, worktree_flow.py, kanban_session_manager.py, etc.) — it is an `orc` error.
   - If the traceback shows the agent produced invalid output, tests failed, git conflict happened, or the error originates in project files — it is a `project` error.
   - If the agent timed out, stalled, or returned empty output — it is a `project` error (agent struggled with the task).
   - When in doubt, classify as `project` — it is safer to attempt an auto-fix than to escalate prematurely.

3. **For `project` errors — formulate a fix task:**
   - Decide which role should handle the fix:
     - `coder` — code bugs, test failures, implementation errors
     - `architect` — design flaws, missing technical specification
     - `reviewer` — the review itself was flawed, re-review needed
     - `integrator` — merge/cherry-pick conflicts, integration failures
   - Write a detailed fix card with:
     - Section "# 1. Product Requirements": What went wrong and what the fix must achieve. Include acceptance criteria.
     - Section "# 2. Technical Design & DoD": Specific files to examine, what to change, a checklist of done criteria.
   - Be specific — include file paths, error messages, line numbers from the traceback.
   - The fix card must be **self-contained**: another agent will read it cold with no prior context.
   - If the error happened in a worktree, reference the worktree path so the fixer knows where to look.

4. **For `orc` errors — write a fix message for the human:**
   - Read the failing ORC module at `{orc_install_path}` to understand the bug.
   - Explain what ORC module failed and why.
   - Point to the specific file and function in `orc_core/`.
   - Suggest a concrete fix (code change, config adjustment, workaround).
   - This message will be sent to Telegram — keep it under 3000 characters.

## Output Contract

Write the incident decision file at: `{decision_path}`

The file MUST have YAML frontmatter with these fields:


- `classification`: `project` or `orc`
- `target_role`: `coder`, `architect`, `reviewer`, or `integrator` (ignored for `orc` classification)
- `fix_title`: one-line title for the fix task (for `project`) or a short summary (for `orc`)
